# BlackVault V1.2 — Wave 1 + Wave 2 Design Spec

**Date:** 2026-03-31
**Branch:** V1.2
**Scope:** Bug fixes, image deletion, global search, range session pagination
**Schema changes:** One additive migration — `ammoStockId` on `RangeSessionAmmoLink` becomes nullable

---

## Overview

This spec covers two waves of changes delivered as a single implementation plan, grouped into four checkpoints to make verification easy.

- **Wave 1** — Bug fixes (no new UI surfaces, targeted edits only)
- **Wave 2** — Three small features: image deletion, global armory search, range session pagination

Password lock and features requiring larger schema changes (maintenance log, range session PDF export) are deferred to a later wave.

## Group 1 — Bug Fixes

### 1a. Add Rounds modal auto-calc (price before quantity)

**File:** `src/app/ammo/page.tsx` — `AddRoundsModal` component

**Problem:** The modal calculates `totalCost = pricePerRound × quantity`. If the user types price-per-round before entering a quantity, the calculation reads stale state (quantity is still 0) and produces 0.

**Fix:** Each `onChange` handler for `pricePerRound` and `quantity` computes the derived field using the *incoming* value rather than reading state. For example, when `pricePerRound` changes to `p`, recalculate `totalCost = p × currentQuantity`. When `quantity` changes to `q`, recalculate `totalCost = currentPricePerRound × q`. This makes order of entry irrelevant.

### 1b. Edit Ammo caliber change — UI re-grouping

**File:** `src/app/ammo/page.tsx` — `handleStockEdit` function

**Problem:** When a stock's caliber is changed via `EditAmmoModal` and saved, the `handleStockEdit` handler updates the stock in place within its original caliber group. The stock appears under the wrong caliber until the page is refreshed.

**Fix:** In `handleStockEdit`, compare `updated.caliber` to the group caliber the stock currently belongs to. If they differ:
1. Remove the stock from the old group; if the group is now empty, remove the group
2. Find or create a group matching `updated.caliber`
3. Insert the updated stock into that group
4. Recalculate `totalQuantity` on both affected groups

### 1c. Delete buttons swallow errors silently

**Files:**
- `src/app/vault/[id]/edit/page.tsx` — firearm delete handler
- `src/app/vault/[id]/builds/[buildId]/page.tsx` — build delete handler

**Problem:** On non-2xx API response, the handlers stop the loading spinner but show no error message. The user sees nothing and may try to navigate away without knowing the deletion failed.

**Fix:** On a failed delete response, display an inline error message near the delete button using the existing red error style (`bg-[#E53935]/10 border border-[#E53935]/30`). Message: *"Failed to delete. Please try again."* The error clears if the user retries.

### 1d. Ammo stock deletion — preserve range session history

**Files:**
- `prisma/schema.prisma` — `RangeSessionAmmoLink.ammoStockId` → `String?`
- `src/app/api/ammo/[id]/route.ts` — DELETE handler
- `src/app/ammo/page.tsx` — delete confirmation UI

**Problem:** Deleting an ammo stock currently cascade-deletes all `RangeSessionAmmoLink` rows that reference it, silently destroying range session history.

**Schema change:** Make `ammoStockId` nullable on `RangeSessionAmmoLink`:
```prisma
ammoStockId String?
ammoStock   AmmoStock? @relation(fields: [ammoStockId], references: [id], onDelete: SetNull)
```
Run `npx prisma migrate dev --name nullable_ammo_stock_link`.

**Delete flow:**
1. Before deleting, count linked `RangeSessionAmmoLink` rows for this stock
2. If count > 0, show confirmation: *"This lot is linked to X range sessions. Sessions will be preserved but will show the ammo as removed. Continue?"*
3. Prisma `onDelete: SetNull` handles nullifying the FK automatically — no manual update step needed
4. In the session history view, ammo links where `ammoStock` is null display as *"Removed lot"*

## Group 2 — Image Deletion

### 2a. New API route

**File:** `src/app/api/images/delete/route.ts` (new)
**Method:** `DELETE`
**Query params:** `filename`, `entityType` (`firearm` | `accessory`), `entityId`

Behaviour:
1. Validate `entityType` is in the allowlist
2. Validate `filename` matches safe pattern (alphanumeric, hyphens, underscores, single dot extension — reuse existing `SAFE_ENTITY_ID` style regex)
3. Resolve the absolute path: `DATA_DIR/uploads/<entityType>s/<filename>`
4. Attempt `fs.unlink` — if file is already gone, continue (do not error)
5. Clear `imageUrl` on the matching Prisma record (`prisma.firearm.update` or `prisma.accessory.update`)
6. Return `{ success: true }`

### 2b. UI — remove image button

**Files:**
- `src/app/vault/[id]/page.tsx`
- Accessory detail page (wherever `imageUrl` is displayed)

A small `×` button overlaid on the image thumbnail, visible on hover (desktop) or always visible (mobile). Clicking shows inline confirmation: *"Remove image?"* with **Yes** / **Cancel**. On confirm, calls the delete route then removes the image from local state so the UI updates immediately without a reload.

## Group 3 — Global Armory Search

### 3a. Search API

**File:** `src/app/api/search/route.ts` (new)
**Method:** `GET`
**Query param:** `q` (string, min 2 chars)

Returns results grouped by entity type. Each result includes `id`, `name` (or display label), `subtitle` (e.g. caliber or type), and `url` (the page to navigate to).

Prisma queries (all sequential, not `Promise.all`):
```ts
prisma.firearm.findMany({
  where: { OR: [
    { name: { contains: q, mode: "insensitive" } },
    { manufacturer: { contains: q, mode: "insensitive" } },
    { model: { contains: q, mode: "insensitive" } },
    { caliber: { contains: q, mode: "insensitive" } },
  ]},
  take: 5,
  select: { id, name, manufacturer, model, caliber, type }
})
// same pattern for accessory, ammoStock (brand/caliber), build (name)
```

Response shape:
```json
{
  "firearms": [...],
  "accessories": [...],
  "ammo": [...],
  "builds": [...]
}
```

Returns `{ firearms: [], accessories: [], ammo: [], builds: [] }` for queries under 2 characters (no results, no DB hit).

### 3b. Search UI

**Files:**
- `src/components/layout/Sidebar.tsx` (or equivalent nav layout file) — add magnifying glass icon button to top-right of header
- `src/components/search/GlobalSearch.tsx` (new client component)

The `GlobalSearch` component renders as a full-screen overlay (`fixed inset-0 z-50`) with a dark backdrop. Contains:
- Auto-focused text input at the top
- Results list below, grouped by category with a header label (FIREARMS / ACCESSORIES / AMMO / BUILDS)
- Each result row shows name + subtitle, is a link that navigates on click
- *"Search firearms, accessories, ammo…"* placeholder when input is empty
- *"No results found"* when query ≥ 2 chars and all groups are empty
- Closes on Escape key or clicking the backdrop

Search is debounced 300ms to avoid firing on every keystroke.

The magnifying glass icon (`Search` from lucide-react) sits in the existing nav header. Clicking it sets a `searchOpen` state in the layout or via a lightweight context.

## Group 4 — Range Session Pagination

### 4a. API change

**File:** `src/app/api/range/sessions/route.ts`

Add `take` and `skip` query params to the `GET` handler:
```ts
const take = Math.min(parseInt(searchParams.get("take") ?? "20"), 50);
const skip = parseInt(searchParams.get("skip") ?? "0");
// apply to prisma.rangeSession.findMany({ take, skip, ... })
```

Default: `take=20, skip=0`. Max enforced at 50. Existing callers that omit both params get the first 20 (a safe default — no caller currently relies on receiving all sessions).

### 4b. Component change

**File:** `src/components/range/RangeWorkspace.tsx`

On mount (when `view === "session-history"`), fetch `/api/range/sessions?take=20&skip=0` and store results in state. Track `offset` (number of sessions loaded so far).

At the bottom of the session list:
- Show *"Showing X sessions"* label
- If the last fetch returned exactly 20, show a **Load more** button
- Clicking **Load more** fetches `?take=20&skip=<offset>`, appends results, updates offset
- When a fetch returns fewer than 20, the **Load more** button is hidden

All existing per-row functionality (expand to view, delete) is unaffected.
