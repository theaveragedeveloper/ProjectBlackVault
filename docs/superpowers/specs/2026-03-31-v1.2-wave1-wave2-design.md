# BlackVault V1.2 ‚Äî Wave 1 + Wave 2 Design Spec

**Date:** 2026-03-31
**Branch:** V1.2
**Scope:** Bug fixes, image deletion, global search, range session pagination
**Schema changes:** One additive migration ‚Äî `ammoStockId` on `RangeSessionAmmoLink` becomes nullable; `@@unique` constraint on that table is removed

---

## Overview

This spec covers two waves of changes delivered as a single implementation plan, grouped into four checkpoints to make verification easy.

- **Wave 1** ‚Äî Bug fixes (no new UI surfaces, targeted edits only)
- **Wave 2** ‚Äî Three small features: image deletion, global armory search, range session pagination

Password lock and features requiring larger schema changes (maintenance log, range session PDF export) are deferred to a later wave.

## Group 1 ‚Äî Bug Fixes

### 1a. Add Rounds modal auto-calc (price before quantity)

**File:** `src/app/ammo/page.tsx` ‚Äî `AddRoundsModal` component

**Problem:** The modal calculates `totalCost = pricePerRound √ó quantity`. If the user types price-per-round before entering a quantity, the calculation reads stale state (quantity is still 0) and produces 0.

**Fix:** Each `onChange` handler for `pricePerRound` and `quantity` computes the derived field using the *incoming* value rather than reading state. For example, when `pricePerRound` changes to `p`, recalculate `totalCost = p √ó currentQuantity`. When `quantity` changes to `q`, recalculate `totalCost = currentPricePerRound √ó q`. This makes order of entry irrelevant.

### 1b. Edit Ammo caliber change ‚Äî UI re-grouping

**File:** `src/app/ammo/page.tsx` ‚Äî `handleStockEdit` function

**Problem:** When a stock's caliber is changed via `EditAmmoModal` and saved, the `handleStockEdit` handler updates the stock in place within its original caliber group. The stock appears under the wrong caliber until the page is refreshed.

**Fix:** In `handleStockEdit`, compare `updated.caliber` to the group caliber the stock currently belongs to. If they differ:
1. Remove the stock from the old group; if the group is now empty, remove the group
2. Find or create a group matching `updated.caliber`
3. Insert the updated stock into that group
4. Recalculate `totalQuantity` on both affected groups

### 1c. Delete buttons swallow errors silently

**Files:**
- `src/app/vault/[id]/edit/page.tsx` ‚Äî firearm delete handler
- `src/app/vault/[id]/builds/[buildId]/page.tsx` ‚Äî build delete handler

**Problem:** On non-2xx API response, the handlers stop the loading spinner but show no error message. The user sees nothing and may try to navigate away without knowing the deletion failed.

**Fix:** On a failed delete response, display an inline error message near the delete button using the existing red error style (`bg-[#E53935]/10 border border-[#E53935]/30`). Message: *"Failed to delete. Please try again."* The error clears if the user retries.

### 1d. Ammo stock deletion ‚Äî preserve range session history

**Files:**
- `prisma/schema.prisma` ‚Äî `RangeSessionAmmoLink.ammoStockId` ‚Üí `String?`, remove `@@unique`
- `src/app/api/ammo/[id]/route.ts` ‚Äî DELETE handler
- `src/app/ammo/page.tsx` ‚Äî delete confirmation UI

**Problem:** Deleting an ammo stock currently cascade-deletes all `RangeSessionAmmoLink` rows that reference it, silently destroying range session history.

**Schema change:** Make `ammoStockId` nullable and switch to `onDelete: SetNull`. Also remove the `@@unique([rangeSessionId, ammoStockId])` constraint ‚Äî once `ammoStockId` is nullable, SQLite treats each NULL as distinct so the constraint no longer enforces the intended invariant. Duplicate non-null links are prevented at the application layer (the existing POST handler already guards this).

```prisma
model RangeSessionAmmoLink {
  id             String        @id @default(cuid())
  rangeSessionId String
  ammoStockId    String?
  roundsUsed     Int
  createdAt      DateTime      @default(now())
  rangeSession   RangeSession  @relation(fields: [rangeSessionId], references: [id], onDelete: Cascade)
  ammoStock      AmmoStock?    @relation(fields: [ammoStockId], references: [id], onDelete: SetNull)
  // @@unique removed ‚Äî see spec note above
}
```

Run `npx prisma migrate dev --name nullable_ammo_stock_link`.

**Delete flow:**
1. Before deleting, count linked `RangeSessionAmmoLink` rows for this stock
2. If count > 0, show confirmation: *"This lot is linked to X range sessions. Sessions will be preserved but will show the ammo as removed. Continue?"*
3. Prisma `onDelete: SetNull` nullifies the FK automatically on delete ‚Äî no manual update step needed
4. In the session history view, ammo links where `ammoStock` is null display as *"Removed lot"*

## Group 2 ‚Äî Image Deletion

### 2a. New API route

**File:** `src/app/api/images/delete/route.ts` (new)
**Method:** `DELETE`
**Query params:** `filename`, `entityType` (`firearm` | `accessory`), `entityId`

The upload route allows `"firearm"`, `"accessory"`, `"ammo"`, and `"build"` entity types, but only firearms and accessories store an `imageUrl` field in the DB. The delete route must restrict to `["firearm", "accessory"]` only. Do NOT reuse the upload route's `ALLOWED_ENTITY_TYPES` constant ‚Äî it has a broader allowlist and will silently permit invalid entity types here.

**Auth and rate limiting:** The delete route must include the same protections as the upload route:
- `const auth = await requireAuth(); if (auth) return auth;`
- `const rateLimit = enforceRateLimit(request); if (rateLimit) return rateLimit;`
- Verify the requesting entity exists (fetch the firearm/accessory record) before deleting the file ‚Äî prevents deleting files for IDs that don't exist

Behaviour:
1. Validate `entityType` is in the local allowlist `["firearm", "accessory"]` only ‚Äî return 400 for any other value
2. Validate `filename` with a filename-specific safe pattern: `/^[a-zA-Z0-9_-]{1,100}\.[a-z0-9]{2,5}$/` ‚Äî this allows the `<entityId>_<timestamp>.<ext>` format that the upload route generates, while blocking path traversal attempts (no slashes, no `..`)
3. Resolve the absolute path using the same `IMAGE_UPLOAD_DIR` environment variable and directory structure that `src/app/api/images/upload/route.ts` uses: `<IMAGE_UPLOAD_DIR>/images/<entityType>s/<filename>` ‚Äî read the upload route's `resolveUploadRoot()` helper to confirm the exact path before implementing
4. Attempt `fs.unlink` ‚Äî if file is already gone (`ENOENT`), continue (do not error)
5. Clear `imageUrl` on the matching Prisma record: `prisma.firearm.update` when `entityType === "firearm"`, `prisma.accessory.update` when `entityType === "accessory"`
6. Return `{ success: true }`

### 2b. UI ‚Äî remove image button

**Files:**
- `src/app/vault/[id]/page.tsx`
- `src/app/accessories/[id]/page.tsx`

A small `√ó` button overlaid on the image thumbnail, visible on hover (desktop) or always visible (mobile). Clicking shows inline confirmation: *"Remove image?"* with **Yes** / **Cancel**. On confirm, calls the delete route then clears the image from local component state so the UI updates immediately without a reload.

## Group 3 ‚Äî Global Armory Search

### 3a. Search API

**File:** `src/app/api/search/route.ts` (new)
**Method:** `GET`
**Query param:** `q` (string, min 2 chars)

Returns results grouped by entity type. Each result includes `id`, `name` (or display label), `subtitle` (e.g. caliber or type), and `url` (the page to navigate to).

**SQLite note:** Prisma on SQLite does not support `mode: "insensitive"` on `contains` ‚Äî it is silently ignored, making search case-sensitive. To work around this, lower-case `q` before querying and rely on SQLite's default case-folding behaviour for ASCII characters. This gives case-insensitive matching for standard Latin characters.

Prisma queries (all sequential ‚Äî no `Promise.all`):
```ts
const q = rawQ.toLowerCase();

// Firearms
const firearms = await prisma.firearm.findMany({
  where: { OR: [
    { name: { contains: q } },
    { manufacturer: { contains: q } },
    { model: { contains: q } },
    { caliber: { contains: q } },
  ]},
  take: 5,
  select: { id: true, name: true, manufacturer: true, model: true, caliber: true, type: true }
});

// Accessories ‚Äî same pattern: name, manufacturer, model, type
// AmmoStock ‚Äî brand, caliber
// Build ‚Äî name only
```

Response shape:
```json
{
  "firearms": [{ "id": "...", "name": "...", "subtitle": "Glock ¬∑ 9mm", "url": "/vault/..." }],
  "accessories": [...],
  "ammo": [...],
  "builds": [...]
}
```

Returns `{ firearms: [], accessories: [], ammo: [], builds: [] }` immediately (no DB hit) when `q.length < 2`.

### 3b. Search UI

**Files:**
- Nav/header layout file (verify actual path ‚Äî check `src/app/layout.tsx` and `src/components/layout/Sidebar.tsx`) ‚Äî add magnifying glass icon button to top-right of header
- `src/components/search/GlobalSearch.tsx` (new client component)

The `GlobalSearch` component renders as a full-screen overlay (`fixed inset-0 z-50`) with a dark backdrop. Contains:
- Auto-focused text input at the top
- Results list below, grouped by category with a header label (FIREARMS / ACCESSORIES / AMMO / BUILDS)
- Each result row shows name + subtitle and navigates on click
- *"Search firearms, accessories, ammo‚Ä¶"* placeholder shown when input is empty (no fetch)
- *"No results found"* when query ‚â• 2 chars and all groups are empty
- Closes on Escape key or clicking the backdrop

The debounced fetch (300ms) only fires when `q.length >= 2` ‚Äî it must not fire on mount or when the input is empty.

**Nav placement ‚Äî two locations required:**
- **Desktop:** Add the `Search` icon button inside `src/components/layout/Sidebar.tsx`, near the top of the sidebar (the desktop layout has no separate header bar ‚Äî the sidebar is the only persistent nav element on desktop)
- **Mobile:** Add the same `Search` icon button inside the `MobileHeader` component (the `md:hidden` header bar used on mobile)

Both buttons set the same `searchOpen` state. The `GlobalSearch` overlay itself is rendered once at the layout level (not duplicated), positioned with `fixed inset-0 z-50` so it covers both layouts.

## Group 4 ‚Äî Range Session Pagination

### 4a. API change

**File:** `src/app/api/range/sessions/route.ts`

Add `take` and `skip` query params to the `GET` handler:
```ts
const take = Math.min(parseInt(searchParams.get("take") ?? "20"), 50);
const skip = parseInt(searchParams.get("skip") ?? "0");
// apply to prisma.rangeSession.findMany({ take, skip, ... })
```

Default: `take=20, skip=0`. Max enforced at 50.

### 4b. Component change

**File:** `src/components/range/RangeWorkspace.tsx`

**Important:** The existing `loadSessions` function is called unconditionally on mount (not gated to `view === "session-history"`) and other parts of the component (drill history, inline delete splice) depend on the full `sessions` array. The pagination must be implemented as a **separate fetch path** for the session-history view, not by modifying the existing `loadSessions` call:

- Keep the existing `loadSessions` call as-is for non-history uses (drill performance, etc.)
- Add a new `sessionHistoryPage` state and a `loadSessionHistoryPage(skip)` function that fetches `?take=20&skip=<skip>` specifically for the history view
- The session-history view maintains its own `historyItems` array (starts empty, populated on first view of session-history tab)
- The existing `handleDeleteSession` function unconditionally splices from `sessions`. For the history view, create a parallel `handleDeleteHistoryItem(id)` function that splices from `historyItems` instead. Wire the delete button in the session-history view to `handleDeleteHistoryItem`, not `handleDeleteSession`
- The session-history view needs its own `historyConfirmId` state (separate from any confirm state used in the log-session view) to track the inline "Delete? Yes / Cancel" confirmation. `handleDeleteHistoryItem` must set `historyConfirmId` to `null` on both success and failure so the confirmation UI dismisses correctly after the operation

At the bottom of the session history list:
- Show *"Showing X sessions"* label
- **Load more** button appears if the last fetch returned exactly 20
- Clicking **Load more** fetches the next page and appends to `historyItems`
- Button is hidden when last fetch returned fewer than 20
