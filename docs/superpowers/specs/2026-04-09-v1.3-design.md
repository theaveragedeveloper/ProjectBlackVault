# BlackVault V1.3 — Design Spec

**Date:** 2026-04-09
**Branch:** V1.3
**Status:** Approved

---

## Overview

V1.3 is a focused release with two features:

1. **Ammo Auto-Deduction: Fix Insufficient Stock Behavior** — deduction already happens automatically via `finalize`; the problem is it hard-blocks and errors when stock runs low instead of warning and continuing
2. **Archive Firearms & Accessories** — preserve history when a gun or accessory is sold/transferred

---

## Feature 1: Fix Ammo Auto-Deduction Insufficient Stock Behavior

### Current Behavior

Ammo deduction is already implemented and runs automatically. After a range session is saved, `POST /api/range/sessions/[id]/finalize` is called immediately by `RangeWorkspace.tsx`. The finalize route decrements `AmmoStock.quantity` and creates an `AmmoTransaction` record of type `RANGE_USE`.

**The problem:** `finalize/route.ts` line 67 contains a hard guard:

```ts
if (stock.quantity < ammoLink.roundsUsed) {
  throw new Error(`Insufficient ammo stock for ${stock.brand} (${stock.caliber})`);
}
```

When this throws, finalize returns a 400 and `RangeWorkspace.tsx` enters `finalizeState === "failed_after_save"`, showing a confusing message: *"Finalize failed. Retry save to safely re-run updates."* The session is saved but no deduction occurred. The user has no clear path forward.

### Desired Behavior

Replace the hard-block with a **warn-and-continue** approach:

- If `roundsUsed > currentStock`, deduct what is available (floor stock at 0), complete finalize normally
- Record the `AmmoTransaction` with the actual quantity deducted (`= currentStock`, not `roundsUsed`)
- After finalize succeeds, the existing `finalizeState === "success"` UI should additionally show a warning if any stock was exhausted: *"⚠ Federal 9mm 115gr stock depleted — you may want to update your inventory"*
- `newQty` on the `AmmoTransaction` record is always 0 in the floored case (not negative)

### What Changes

**`src/app/api/range/sessions/[id]/finalize/route.ts`**

Replace lines 67–69:
```ts
// REMOVE:
if (stock.quantity < ammoLink.roundsUsed) {
  throw new Error(`Insufficient ammo stock for ${stock.brand} (${stock.caliber})`);
}
```

With:
```ts
const deductQty = Math.min(ammoLink.roundsUsed, stock.quantity);
const wasInsufficient = ammoLink.roundsUsed > stock.quantity;
```

Use `deductQty` for the actual update and transaction record. Collect `wasInsufficient` cases and include them in the route's response JSON as `{ stockWarnings: [{ name, caliber, shortfall }] }`.

**`src/components/range/RangeWorkspace.tsx`**

After a successful finalize (`finalizeRes.ok`), check `finalizeJson.stockWarnings`. If any exist, set the existing `warning` state with a message listing the depleted stocks. The success state still shows (`finalizeState === "success"`), with the warning displayed below it.

The `failed_after_save` error path for insufficient stock is no longer reachable from this cause. Other finalize errors (e.g., "Range session not found") still use the existing error path.

### Error Responses

- `404` — session not found (unchanged)
- `400` — other unexpected errors (unchanged)
- `200` with `stockWarnings: []` — normal success (unchanged shape, just adds the warnings array)
- `200` with `stockWarnings: [...]` — success with depletion warnings

## Feature 2: Archive Firearms & Accessories

### Problem

There is no way to retire a firearm or accessory from active inventory without deleting it. Deletion destroys all history (round counts, maintenance logs). Users who sell or transfer a gun want to preserve the record.

### Behavior

**Archiving:**
- An "Archive" button appears on the firearm and accessory detail pages, in the actions/danger zone section (between the Edit and Delete buttons)
- Clicking Archive shows an inline confirmation: "Archive this firearm? It will be hidden from your active vault but all history is preserved."
- On confirm, the item is archived (`archivedAt` set to current timestamp)

**What archiving affects:**
- Firearm/Accessory hidden from the main Vault and Accessories list pages
- Associated builds hidden from the Builds page (browseable UI only — see note on range sessions below)
- Round count logs, maintenance logs, battery logs: preserved and visible when the archived record is expanded
- Range sessions: **unaffected** — sessions remain fully active and visible in Range History, with the archived firearm's name still shown. Build IDs on sessions use `onDelete: SetNull` in the schema so no FK issues arise.
- Archived firearm's builds are hidden from the range session logging dropdowns in `RangeWorkspace.tsx` (do not show archived-firearm builds in the firearm/build selectors)
- Full Armory export (`/api/exports/full-armory`): add `archivedAt: null` filter so archived items do not appear in exports

**Archived section on list pages:**
- Bottom of `/vault` and `/accessories` pages: collapsed "ARCHIVED (N)" section, toggled by chevron
- Collapsed by default
- Each archived card shows: name, caliber/type, archived date, and an "Unarchive" link
- Clicking Unarchive restores the item immediately (no confirmation required — non-destructive)

### Schema

Add `archivedAt DateTime?` to both models:

```prisma
model Firearm {
  // ... existing fields
  archivedAt DateTime?
}

model Accessory {
  // ... existing fields
  archivedAt DateTime?
}
```

Run: `npx prisma migrate dev --name add-archived-at` in development.
The production Dockerfile entrypoint uses `npx prisma migrate deploy` — this migration will be applied automatically on container start.

### API

**Filter active items** — add `archivedAt: null` to existing queries:

| Location | Change |
|---|---|
| `GET /api/firearms` | `where: { archivedAt: null }` by default. Also add `?archived=true` query param support: when present, return `where: { archivedAt: { not: null } }` instead. Both cases handled in the same route. |
| `GET /api/accessories` | `where: { archivedAt: null }` |
| `src/app/builds/page.tsx` — `getAllBuilds()` | `where: { archivedAt: null }` on the Firearm query (this is a direct Prisma call, not via API route) |
| `GET /api/exports/full-armory` | `where: { archivedAt: null }` on both firearm and accessory queries |
| `RangeWorkspace.tsx` firearm/build selectors | No component change needed — once `GET /api/firearms` defaults to active-only, the dropdowns are automatically filtered. |

**New archive endpoints:**

`PATCH /api/firearms/[id]/archive`
```
Body:     { archived: boolean }
Response: { success: true }
Errors:   404 if not found, 500 on DB error
```
Sets `archivedAt` to `new Date()` when `archived: true`, `null` when `archived: false`.

`PATCH /api/accessories/[id]/archive`
Same shape as above.

### UI Changes

**`src/app/vault/[id]/page.tsx`**
Add "Archive" button in the actions section. On click, show inline confirmation. On confirm, call `PATCH /api/firearms/[id]/archive` with `{ archived: true }`, then redirect to `/vault`.

**`src/app/accessories/[id]/page.tsx`**
Same pattern — call `PATCH /api/accessories/[id]/archive`, redirect to `/accessories`.

**`src/app/vault/page.tsx`** (client component — fetches via `fetch("/api/firearms")`)
- The default `fetch("/api/firearms")` now returns active-only (filter added in step 2 above)
- Make a second `fetch("/api/firearms?archived=true")` for the collapsed archived section
- Render collapsed "ARCHIVED (N)" section at the bottom with chevron toggle

**`src/app/accessories/page.tsx`** (server component — uses Prisma directly)
- Add a second Prisma query: `prisma.accessory.findMany({ where: { archivedAt: { not: null } } })`
- Render collapsed "ARCHIVED (N)" section at the bottom

**`src/app/builds/page.tsx`**
Filter: update `getAllBuilds()` to only return builds belonging to non-archived firearms.

## Implementation Order

1. **Ammo finalize fix** (no schema change, self-contained)
   - Modify `finalize/route.ts`: replace hard-block with floor + `stockWarnings` in response
   - Modify `RangeWorkspace.tsx`: display stock depletion warning on finalize success

2. **Archive — Schema & API**
   - Prisma migration (`archivedAt` on Firearm + Accessory)
   - Add `PATCH /api/firearms/[id]/archive` and `PATCH /api/accessories/[id]/archive`
   - Update `GET /api/firearms` to support `?archived=true` param
   - Update `GET /api/accessories`, `GET /api/exports/full-armory`, and `getAllBuilds()` with `archivedAt: null` filter

3. **Archive — UI**
   - Archive button + confirmation on firearm and accessory detail pages
   - Collapsed archived section on `/vault` and `/accessories` pages
   - Filter archived-firearm builds from RangeWorkspace dropdowns

---

## Out of Scope for V1.3

- Range session export / PDF reports
- Password / PIN lock
- Drill analytics beyond what already exists
- "Sold" vs "Archived" status distinction (single `archivedAt` field is sufficient)
- Archiving builds independently of their parent firearm
