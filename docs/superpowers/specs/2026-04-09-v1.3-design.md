# BlackVault V1.3 — Design Spec

**Date:** 2026-04-09
**Branch:** V1.3
**Status:** Approved

---

## Overview

V1.3 is a focused release with two features:

1. **Ammo Auto-Deduction** — close the loop between range session logging and ammo inventory
2. **Archive Firearms & Accessories** — preserve history when a gun or accessory is sold/transferred

---

## Feature 1: Ammo Auto-Deduction from Range Sessions

### Problem

When a range session is logged with ammo linked, the ammo stock quantity is never decremented. Users must manually update their stock separately, which is easy to forget.

### Behavior

After a range session is successfully saved, a confirmation modal fires asking the user whether to deduct rounds from the linked ammo stock.

**Normal flow:**
1. User fills out and saves a range session (with ammo linked and rounds fired)
2. Session commits to DB successfully
3. Confirmation modal appears: "Deduct X rounds from [Ammo Name]?"
   - Shows current stock and resulting stock after deduction
   - Two actions: **"Yes, deduct"** and **"Skip"**
4. On confirm: stock is decremented, success toast shown ("Stock updated")
5. On skip: session saved, stock unchanged — no further prompts

**Warning state (rounds exceed stock):**
- If `roundsFired > currentStock`, the modal still appears but shows a warning:
  - "⚠ Only X rounds in stock — deducting will set stock to 0"
- User can still confirm (stock floors at 0) or skip
- Stock does not go negative — it floors at 0

**Edge cases:**
- If no ammo is linked to the session, modal does not appear
- If multiple ammo stocks are linked (future), a row per stock is shown in the modal
- If `roundsFired` is 0 or null, modal does not appear

### Schema

No schema changes required. `RangeSessionAmmoLink` and `AmmoStock.quantity` already exist.

### API

**New endpoint:** `POST /api/ammo/[id]/deduct`

```
Body: { rounds: number, sessionId: string }
Response: { success: true, newQuantity: number }
```

- Decrements `AmmoStock.quantity` by `rounds`, floored at 0
- Validates that `sessionId` exists and is linked to this ammo stock
- Returns 404 if ammo stock not found
- Returns 400 if `rounds` is not a positive integer

### UI Changes

- `src/components/range/RangeWorkspace.tsx` — add deduction confirmation modal that fires after `handleSaveSession()` resolves successfully
- Modal is inline (no route change), styled to match existing vault modals
- State: `deductionPending: { ammoId, ammoName, rounds, currentStock } | null`

## Feature 2: Archive Firearms & Accessories

### Problem

There is no way to retire a firearm or accessory from active inventory without deleting it. Deletion destroys all history (round counts, maintenance logs). Users who sell or transfer a gun want to preserve the record.

### Behavior

**Archiving:**
- An "Archive" button appears on the firearm and accessory detail pages, in the actions/danger zone section (visually between Edit and Delete)
- Clicking Archive shows a brief inline confirmation: "Archive this firearm? It will be hidden from your active vault but all history is preserved."
- On confirm, the item is archived (timestamp recorded)

**What archiving affects:**
- Firearm/Accessory hidden from the main Vault and Accessories list pages
- Associated builds hidden from the Builds page
- Round count logs, maintenance logs, battery logs: preserved and visible when the archived record is expanded
- Range sessions: **unaffected** — sessions remain fully active and visible in Range History, with the archived firearm's name still shown

**Archived section on list pages:**
- Bottom of `/vault` and `/accessories` pages: collapsed "ARCHIVED (N)" section
- Collapsed by default, toggled with a chevron
- Each archived card shows: name, caliber/type, archived date, and an "Unarchive" button
- Unarchiving restores the item to active status immediately

**Unarchiving:**
- Unarchive button on each archived card
- No confirmation required (non-destructive)
- Item reappears in active list

### Schema

Add `archivedAt DateTime?` to both `Firearm` and `Accessory` models:

```prisma
model Firearm {
  // ... existing fields
  archivedAt DateTime?
}

model Accessory {
  // ... existing fields
  archivedAt DateTime?
}
```

Run `npx prisma migrate dev --name add-archived-at`.

### API

**Filter active items** — all existing list endpoints add `where: { archivedAt: null }`:
- `GET /api/firearms` — add `archivedAt: null` filter
- `GET /api/accessories` — add `archivedAt: null` filter
- `GET /api/builds` — add `firearm: { archivedAt: null }` filter

**New archive endpoints:**

`PATCH /api/firearms/[id]/archive`
```
Body: { archived: boolean }
Response: { success: true }
```
Sets `archivedAt` to `new Date()` when archiving, `null` when unarchiving.

`PATCH /api/accessories/[id]/archive`
Same shape as above.

### UI Changes

- `src/app/vault/[id]/page.tsx` — add Archive button in actions section
- `src/app/accessories/[id]/page.tsx` — add Archive button in actions section
- `src/app/vault/page.tsx` — add collapsed Archived section at bottom; fetch archived items separately
- `src/app/accessories/page.tsx` — add collapsed Archived section at bottom; fetch archived items separately
- `src/app/builds/page.tsx` — filter query to exclude builds belonging to archived firearms

---

## Implementation Order

1. Ammo Auto-Deduction (self-contained, no schema change)
   - Add `POST /api/ammo/[id]/deduct` route
   - Add deduction modal to `RangeWorkspace.tsx`

2. Archive — Schema & API
   - Prisma migration (`archivedAt` on Firearm + Accessory)
   - Add archive/unarchive PATCH endpoints
   - Update list endpoints to filter archived items

3. Archive — UI
   - Archive button on detail pages
   - Collapsed archived section on list pages
   - Filter builds page

---

## Out of Scope for V1.3

- Range session export / PDF reports
- Password / PIN lock
- Drill analytics beyond what already exists
- "Sold" vs "Archived" status distinction (single `archivedAt` field is sufficient)
- Archiving builds independently of their parent firearm
